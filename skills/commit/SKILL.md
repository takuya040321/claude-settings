# Commit

Conventional Commits形式に基づくコミットメッセージを作成する。

## コミットメッセージ形式

```
<type>: <subject>

<body>
```

- **type**: 英語プレフィックス（必須）
- **subject**: 日本語で簡潔に（必須、50文字以内、文末に句点なし）
- **body**: 日本語で補足説明（任意・簡潔に）

## Type一覧

| Type | 説明 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（空白、フォーマット等） |
| `refactor` | バグ修正や機能追加ではないコード変更 |
| `perf` | パフォーマンス改善 |
| `test` | テストの追加・修正 |
| `build` | ビルドシステムや外部依存に関する変更 |
| `ci` | CI設定ファイルやスクリプトの変更 |
| `chore` | その他の変更（ソースやテストの変更を含まない） |
| `revert` | 以前のコミットの取り消し |

## 例

```bash
feat: ユーザー認証機能を追加

fix: ログイン時のnullポインタ例外を修正

fix: セッションタイムアウトを修正

トークン更新処理が正しく動作していなかった

refactor: データベース接続処理を共通化

docs: APIリファレンスを更新

test: ユーザーサービスのテストを追加
```

## コミット手順

1. `git status`で変更ファイルを確認
2. `git diff`でstaged/unstaged変更を確認
3. `git log`で直近のコミットスタイルを確認
4. 変更内容を分析し、適切なtypeを選択
5. 日本語でsubjectを作成（「〜する」「〜を追加」形式）
6. HEREDOCでコミット実行：
   ```bash
   git commit -m "$(cat <<'EOF'
   type: subject
   EOF
   )"
   ```

## コミット粒度

### 基本原則

**「そのコミットだけをrevertしても、他の機能が壊れない単位」** で分ける。

- 1つの機能追加（関連するフロント・バック・DBマイグレーションを含む）
- 1つのバグ修正
- 1つのリファクタリング（例：関数の抽出、命名変更）
- 1つの設定変更

```bash
# 良い例：1コミット1目的
feat: ユーザー登録APIを追加
fix: メールアドレスのバリデーションを修正

# 悪い例：複数の目的が混在
feat: ユーザー登録APIを追加し、ログイン画面のCSSも修正
```

### サイズの目安

| 指標 | 目安 | 補足 |
|------|------|------|
| 変更行数 | ~300行以内 | これを超えたら分割を検討 |
| ファイル数 | ~10ファイル以内 | 関連ファイルならまとめてOK |
| レビュー時間 | ~30分で理解できる | レビュアー視点での判断基準 |

自動生成ファイル（lockファイル、マイグレーション等）は行数に含めない。

### 複数ファイルにまたがる変更

**目的が同じならまとめる、違うなら分ける。**

```bash
# まとめてOK：1つの機能に関連するファイル群
git commit -m "feat: ユーザー検索機能を追加"
# → controller, service, repository, route, migration を含む

# 分けるべき：独立した変更
git commit -m "fix: ログイン時のエラーハンドリングを修正"
git commit -m "style: フッターのレイアウトを調整"
```

### リファクタリングと機能追加は必ず分離

```bash
# Step 1: まずリファクタリング（動作は変えない）
git commit -m "refactor: 認証処理をAuthServiceに抽出"

# Step 2: その上に機能追加
git commit -m "feat: OAuth2ログインを追加"
```

### テストコードの扱い

| ケース | 方針 |
|--------|------|
| 新機能 + そのテスト | **同じコミット** |
| バグ修正 + 回帰テスト | **同じコミット** |
| 既存コードへのテスト追加のみ | **別コミット**（`test:`） |
| テストのリファクタリング | **別コミット**（`refactor:` or `test:`） |

## 注意事項

- 1コミット = 1つの論理的な変更
- .env等の機密ファイルはコミットしない
- WIPコミットは避ける
- `git commit --amend`は直前の自分のコミットかつ未pushの場合のみ使用
