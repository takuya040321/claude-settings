# マークダウンライター

技術ドキュメントを美しく構造化されたマークダウンで作成・整形するスキル。

## 基本原則

### 可読性を最優先

- 一目で構造が把握できること
- 適切な空白行で視覚的に区切る
- 長すぎる行は避ける（80-100文字目安）

### 一貫性を保つ

- 見出しレベルは論理的に使用
- リストのスタイルを統一
- コードブロックの言語を明示

## ドキュメント構造

### 見出しの階層

```markdown
# ドキュメントタイトル（H1は1つだけ）

## 大セクション（H2）

### 中セクション（H3）

#### 小セクション（H4）
```

**ルール**：
- H1はドキュメントに1つだけ
- 見出しレベルは飛ばさない（H2の次はH3）
- 見出しの前後に空行を入れる

### 目次の構造

長いドキュメントには目次を追加：

```markdown
## 目次

- [セクション1](#セクション1)
  - [サブセクション1.1](#サブセクション11)
  - [サブセクション1.2](#サブセクション12)
- [セクション2](#セクション2)
```

## テキストフォーマット

### 強調

| 記法 | 用途 | 例 |
|------|------|-----|
| `**太字**` | 重要な用語、キーワード | **必須**、**注意** |
| `*斜体*` | 英語の強調、書籍名 | *italic*, *Design Patterns* |
| `` `コード` `` | コード、コマンド、ファイル名 | `npm install`, `config.json` |

### リスト

**箇条書き**（順序なし）：
```markdown
- 項目1
- 項目2
  - ネスト項目2.1
  - ネスト項目2.2
- 項目3
```

**番号付き**（順序あり）：
```markdown
1. 最初のステップ
2. 次のステップ
3. 最後のステップ
```

**使い分け**：
- 順序が重要 → 番号付き
- 順序が関係ない → 箇条書き

## 表の作成

### 基本構文

```markdown
| ヘッダー1 | ヘッダー2 | ヘッダー3 |
|-----------|-----------|-----------|
| データ1   | データ2   | データ3   |
| データ4   | データ5   | データ6   |
```

### 整列

```markdown
| 左揃え | 中央揃え | 右揃え |
|:-------|:--------:|-------:|
| Left   | Center   | Right  |
```

### 表を使うべき場面

- 比較情報（機能比較、オプション比較）
- 属性と値のペア（設定項目一覧）
- マトリクス形式のデータ

## コードブロック

### インラインコード

文中のコード、コマンド、ファイル名には `` ` `` を使用：

```markdown
`npm install`コマンドを実行し、`package.json`を確認してください。
```

### コードブロック

言語を必ず指定：

````markdown
```python
def hello():
    print("Hello, World!")
```
````

### 主な言語指定

| 言語 | 指定 |
|------|------|
| Python | `python` |
| JavaScript | `javascript` または `js` |
| TypeScript | `typescript` または `ts` |
| Bash/Shell | `bash` または `sh` |
| JSON | `json` |
| YAML | `yaml` |
| SQL | `sql` |
| Markdown | `markdown` |

## 技術ドキュメント別テンプレート

### README

```markdown
# プロジェクト名

簡潔な説明（1-2文）

## 機能

- 機能1
- 機能2

## インストール

\`\`\`bash
npm install package-name
\`\`\`

## 使い方

\`\`\`javascript
import { something } from 'package-name';
\`\`\`

## 設定

| オプション | 型 | デフォルト | 説明 |
|------------|-----|-----------|------|
| option1    | string | "default" | 説明 |

## ライセンス

MIT
```

### API仕様書

```markdown
# API名

## エンドポイント

### GET /api/resource

リソースの一覧を取得する。

**リクエスト**

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| page      | int | No   | ページ番号 |
| limit     | int | No   | 取得件数 |

**レスポンス**

\`\`\`json
{
  "data": [],
  "total": 100,
  "page": 1
}
\`\`\`

**ステータスコード**

| コード | 説明 |
|--------|------|
| 200    | 成功 |
| 400    | リクエスト不正 |
| 500    | サーバーエラー |
```

### 設計書

```markdown
# 機能名 設計書

## 概要

機能の目的と概要を記述。

## アーキテクチャ

### コンポーネント構成

\`\`\`
+-------------+     +-------------+
| Component A |---->| Component B |
+-------------+     +-------------+
\`\`\`

## データモデル

### テーブル: users

| カラム | 型 | 制約 | 説明 |
|--------|-----|------|------|
| id     | int | PK   | ID   |
| name   | varchar(100) | NOT NULL | 名前 |

## シーケンス

1. ユーザーがリクエストを送信
2. サーバーが認証を確認
3. データベースからデータを取得
4. レスポンスを返却
```

## 要件定義書の整形

`requirements-definition`スキルからの出力を整形する際の追加ガイドライン：

### 構造化のポイント

1. **プロジェクト概要**は簡潔に（長くなりすぎない）
2. **機能要件**は表形式の一覧＋詳細の2段構成
3. **非機能要件**はカテゴリ別に整理
4. **優先度**は色分けまたは記号で視覚的に区別

### 機能要件一覧の例

```markdown
## 機能要件一覧

| ID | 機能名 | 概要 | 優先度 |
|----|--------|------|--------|
| FR-001 | ユーザー登録 | 新規ユーザーがアカウントを作成 | 必須 |
| FR-002 | ログイン | メール/パスワードで認証 | 必須 |
| FR-003 | プロフィール編集 | ユーザー情報の更新 | 重要 |
```

### 優先度の表現

```markdown
- **必須** - システム稼働に不可欠
- **重要** - 初期リリースに含めたい
- **あれば良い** - 将来のバージョンで検討
```

## 品質チェックリスト

ドキュメント作成後、以下を確認：

- [ ] H1が1つだけか
- [ ] 見出しレベルが論理的か
- [ ] 空行が適切に入っているか
- [ ] コードブロックに言語が指定されているか
- [ ] 表の列が揃っているか
- [ ] リンクが機能するか
- [ ] 長すぎる行がないか

## 自動連携

`requirements-definition`スキル完了後、このスキルが自動的に呼び出される。

**受け取る入力**：
- 構造化された要件定義情報
- プロジェクト名
- 機能要件リスト
- 非機能要件リスト

**出力**：
- 整形されたマークダウンドキュメント
- 目次付き
- 表形式の要件一覧
